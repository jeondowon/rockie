# Rockie 개발 계획

현재 MVP(마우스 추적 + 말풍선 + 활성 앱 반응)에 이어 붙일 기능들을 정리한 문서입니다.
아키텍처 세부는 `README.md` 참고.

---

## 1. 채팅 기능 (로컬 LLM 대화)

캐릭터를 클릭/입력으로 부르면 짧고 캐주얼하게 대화하는 기능.
무거운 추론이 아니라 **가벼운 캐릭터 채팅**이 목적이므로 경량 모델로 충분하다.

### 1.1 추론 엔진: `node-llama-cpp` (Ollama 미사용)

**결론: 배포용 앱에는 Ollama 의존 대신 `node-llama-cpp`를 내장한다.**

- Ollama 방식은 사용자가 별도 프로그램을 설치해야 해서 배포 허들이 크다.
- `node-llama-cpp`는 llama.cpp를 Node.js 바인딩으로 감싼 라이브러리 → Electron 앱에 npm 패키지로 그대로 포함 가능.
- 사용자는 별도 설치 없이 앱 실행 파일 안에 추론 엔진이 들어가는 구조.
- 모델 가중치(GGUF)만 앱과 함께 배포하거나, **최초 실행 시 다운로드**하도록 구현.

### 1.2 모델 선택

3B급 Instruct 모델이 데스크톱 펫 용도로 최적. 짧고 캐주얼한 반응만 필요.

| 후보                            | 특징                                                              |
| ------------------------------- | ----------------------------------------------------------------- |
| **Qwen2.5 3B Instruct** (1순위) | 다국어(한국어 포함) 성능이 좋아 캐릭터가 한국어로 자연스럽게 답함 |
| Llama 3.2 3B Instruct           | 가볍고 안정적, GGUF 생태계 지원 넓음                              |
| Gemma 3 1B / 4B                 | 1B는 초경량·초고속, 4B는 품질 한 단계 상승                        |

- **양자화: `Q4_K_M` 기본** — 품질/용량 균형이 가장 좋음 (3B 기준 약 2GB 내외).
- 저사양 대비: **Gemma 3 1B를 기본값**으로 두고, 사양 되는 사용자에게 3B 업그레이드 옵션을 주는 구조도 검토.

### 1.3 성능 체감

- 3B Q4_K_M은 GPU 없이 일반 노트북 CPU에서도 초당 몇~십몇 토큰 → 짧은 말풍선 답변용으로 체감 지연 크지 않음.
- 토큰을 **스트리밍**으로 받아 말풍선에 타이핑되듯 흘려보내면 지연 체감이 더 줄어듦.

### 1.4 배포 전략

- 앱 용량 부담(2GB 내외 모델)을 줄이려면 **최초 실행 시 모델 다운로드** 방식 권장 → 인스톨러를 가볍게 유지.
- 다운로드 진행률 UI + 실패/재시도 처리 필요.
- 모델 파일은 `app.getPath("userData")` 하위에 저장 (앱 번들에 넣지 않음).

## 2. 채팅 기능 구현 설계 (기존 구조에 맞춰)

기존 IPC 브릿지(`preload.js`의 `window.petAPI`)와 말풍선(`renderer.js`의 `showBubble`)을 재사용한다.

### 2.1 메인 프로세스 (`main.js` 또는 신규 `llm.js`)

- LLM 로딩/추론은 무거우므로 **메인 프로세스에서 처리**하고 결과만 렌더러로 전달.
  - 로딩 블로킹이 문제되면 `utilityProcess` 또는 워커로 분리 검토.
- 신규 IPC 핸들러:
  - `chat:send` (renderer → main): 사용자 메시지 수신 → 모델 추론 시작.
  - `chat:token` (main → renderer): 스트리밍 토큰 조각 전달.
  - `chat:done` / `chat:error`: 완료·오류 통지.
  - `model:download-progress`: 최초 다운로드 진행률.
- 대화 맥락 유지: 최근 N턴을 메모리에 들고 시스템 프롬프트에 캐릭터 성격을 고정.

### 2.2 preload (`preload.js`)

`petAPI`에 다음 추가:

```js
sendChat: (text) => ipcRenderer.send("chat:send", text),
onChatToken: (cb) => ipcRenderer.on("chat:token", (_e, d) => cb(d)),
onChatDone:  (cb) => ipcRenderer.on("chat:done",  (_e, d) => cb(d)),
onModelDownloadProgress: (cb) => ipcRenderer.on("model:download-progress", (_e, d) => cb(d)),
```

### 2.3 렌더러 (`renderer.js` / `index.html` / `style.css`)

- 입력 UI: 캐릭터 클릭 시 작은 입력창(텍스트 필드)을 캐릭터 근처에 띄움.
  - 입력창이 뜨는 동안은 창이 클릭을 받아야 하므로 `setIgnoreMouseEvents(false)` 유지 필요
    (현재는 `mouseenter/leave`로만 토글 → 입력 포커스 중 통과 방지 로직 추가).
- 출력: `chat:token`을 받아 `showBubble`을 스트리밍 갱신(말풍선 duration 연장/고정).
- 긴 답변 대비 말풍선 크기·줄바꿈·최대 높이 스타일 보강.

### 2.4 캐릭터 페르소나

- 시스템 프롬프트로 "대장님"을 부르는 비서/펫 캐릭터 톤 고정 (짧게, 친근하게, 한국어).
- 기존 `clickReactions`·활성 앱 규칙과 톤을 일치시켜 일관성 유지.

## 3. 구현 순서 (제안)

1. `node-llama-cpp` 설치 + 단일 프롬프트 추론 PoC (모델 경로 하드코딩).
2. IPC 스트리밍 파이프라인 연결 (`chat:send` → `chat:token` → 말풍선 출력).
3. 캐릭터 클릭 → 입력창 UI + 클릭 통과 처리 보정.
4. 대화 맥락(최근 N턴) + 캐릭터 시스템 프롬프트 적용.
5. 최초 실행 시 모델 다운로드 + 진행률 UI.
6. 저사양 대비 모델 선택(1B/3B) 옵션 (트레이 메뉴 또는 설정).
7. 패키징(`electron-builder` 등)에서 네이티브 바인딩 포함 검증.

---

## 4. 열린 이슈 / 확인 필요

- `node-llama-cpp` 네이티브 바인딩이 `electron-builder` 패키징·코드사이닝·공증(notarization)과 충돌 없는지 확인.
- 모델 로딩 중 앱 시작 지연 → 로딩 상태 표시 및 지연 로딩 전략.
- 메모리 사용량(3B Q4_K_M 상시 상주) — 유휴 시 언로드 정책 검토.
- 오프라인 최초 실행(다운로드 불가) 시 폴백 UX.
